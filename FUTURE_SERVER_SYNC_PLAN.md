# 将来のサーバー連携・データ共有に関する設計指針

このドキュメントは、CareMemoアプリにおける将来的な多端末間データ共有（N:1）および自前サーバー（Spring Boot + PostgreSQL）との連携に向けた設計アイデアを整理したものです。

## 1. 現状と課題
### 現状
* 記録者：被記録者 ＝ 1：1（スタンドアロン運用）
* ID管理：Roomデータベースによる自動採番（Int/Long型）
* 動作環境：オフライン完結

### 課題
* 記録者：被記録者 ＝ N：1 への拡張（家族や介護スタッフ間での共有）
* データのマージ：複数端末で生成されたデータの重複・衝突回避
* IDの整合性：端末ごとに独立して発番されるIDの管理

## 2. 実装アイデア：Local-First アプローチ
オフラインでの利便性（Room）を維持しつつ、オンライン時にサーバー（PostgreSQL）と同期する「Local-First」設計を採用する。

### A. ID管理の改善
各Entityに、ローカルの自動採番IDとは別に、世界で唯一無二となる識別子を付与する。
* **UUIDの採用**: `java.util.UUID` を使用し、アプリ側でデータ生成時に文字列のIDを発番する。
* **メリット**: サーバーからの返答を待たずにRoomへ保存でき、かつ多端末間でIDが衝突しない。サーバー側もこのUUIDを主キーまたはユニークキーとして扱うことで、辞書テーブルを介さずに直接紐付けが可能になる。

### B. Entityへの属性追加（同期の三種の神器）
既存のEntityに以下のフィールドを直接付与し、状態を管理する。
1. `uuid: String`: サーバーと同期するための唯一無二の鍵
2. `syncStatus: Int`: 同期フラグ（0:未同期, 1:同期済み）
3. `updatedAt: Long`: 最終更新タイムスタンプ（コンフリクト解決用）
4. `authorId: String`: 記録作成者の識別子

### C. 同期ロジック
1. **保存時**: `syncStatus = 0`（未同期）としてRoomに保存。
2. **オンライン時**: `syncStatus = 0` のデータを抽出してREST API経由でSpring Bootサーバーへ送信。
3. **サーバー処理**: PostgreSQL側で「同一UUID（または同一人物+同一日時）があれば上書き（UPSERT）」を実行。
4. **完了時**: サーバーから成功レスポンスを受け取ったら、Roomの `syncStatus` を `1` に更新。

## 3. インフラ・セキュリティ構成
* **Backend**: Spring Boot (Kotlin)
* **Database**: PostgreSQL
* **Proxy**: Nginx（リバースプロキシとしてSSL終端、APIフォワードを担当）
* **Security**: 
    * HTTPS（Let's Encrypt）による通信暗号化
    * JWT（JSON Web Token）による認証
    * 行レベルのセキュリティ（自分の権限があるデータのみ参照可能）

## 4. 移行ステップ
1. Androidアプリ側のEntity IDをUUID（文字列）ベースに移行・準備する。
2. Spring Bootで基礎的なログイン・CRUD APIを作成する。
3. Roomに `syncStatus` を導入し、バックグラウンドでの同期処理を実装する。
