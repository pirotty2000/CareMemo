# 簡易的なマルチ・デバイス利用の構想（案）

## 1. コンセプト
専用のサーバーを構築せず、運用ルールとファイルの受け渡し（Google Drive等）によって、複数の端末間でデータを同期・共有する。

## 2. 運用ルールと役割分担
- **マスター端末（管理者）**: `Person`（利用者）データの新規追加・基本情報の変更を行う権限を持つ。
- **一般端末（スタッフ）**: マスターから配布された利用者データを取り込み、日々の記録（詳細データ）を作成する。
- **ID管理**: `Person` の主キー（ID）はマスター端末の発行したものを固定で利用し、一般端末では新規発行を行わないことで重複を回避する。

## 3. データ構造と同期ロジック
- **詳細データの主キー（UUID）**: 記録（メモ等）の主キーには `UUID` (String) を採用し、複数端末で作成された記録がマージ時に衝突しないようにする。
- **論理削除の導入**: データの削除は物理削除ではなく、`isDeleted` フラグによる論理削除とする。これにより、ある端末での削除操作を他の端末へ伝播させる。
- **最終更新日時の保持**: `updatedAt` カラムを持ち、マージ時に同じUUIDのデータが存在する場合は、日時の新しい方を採用（Upsert）する。

## 4. 日々の運用フロー
1. **業務終了後**: 各スタッフが自分の端末で作成した詳細データをJSON形式でエクスポートし、クラウドストレージ（Google Drive等）へ保存する。
   - ファイル名は `CareMemo_日付_スタッフ名.json` のように重複しないルールとする。
2. **業務開始前**: 各スタッフが、クラウド上に保存された前日分（全員分）のJSONファイルを一括でインポートし、自端末のDBを最新の状態にマージする。

## 5. 今後の検討事項（要深掘り）
1. **マージロジックの実装**: 
   - 現在の「全削除して置換」から「UUIDをキーとしたUpsert」へのインポート処理の書き換え。
2. **データ移行プラン**: 
   - 既存の `Long` 型IDデータをどうやって安全に `UUID` へ移行するか。
   - 既存のバックアップファイル（JSON）を新形式へ変換するツールの要否。
3. **利用者（Person）の削除ルール**: 
   - 管理者が利用者を「利用終了」または「完全抹消」した際、他端末へどう安全に伝えるか。
4. **不整合の検知**: 
   - 端末間の時刻設定のズレによる `updatedAt` の逆転現象への対策。
5. **UI/UXの改善**: 
   - 複数ファイルを一括選択してインポートする機能。
   - 現在の同期状態（いつ、誰のデータまで取り込んだか）の可視化。
